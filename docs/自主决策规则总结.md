# 自主决策规则总结

> 完整英文版规则见 `AUTONOMOUS_DECISION_RULES.md`

## 核心理念

在 `CC_MODE=autonomous` 模式下，agent 应该**完全自主**地推进任务，只在真正卡住时才升级。

**关键原则**：
- 所有编程相关的操作都自动批准
- 安全通过沙箱、版本控制保证，不通过交互确认
- 人类选择自主模式 = 完全信任 agent

---

## 快速决策表

| 情况 | 决策 | 升级条件 |
|------|------|---------|
| **Yes/No 确认** | 解析提示格式，选择代表"同意/继续"的选项（可能是 y、1、a 等） | 从不升级（除非问"是否放弃任务"） |
| **多项选择** | 选推荐/默认选项，或选代表"继续"的选项，或选第一个 | 从不升级 |
| **光标导航** | 自动导航并确认 | 从不升级 |
| **开放性问题** | 使用合理默认值/标准实践 | 只在需要真实外部信息时升级 |
| **遇到错误** | 自动修复，最多尝试 2 次 | 同一错误第 3 次出现时升级 |
| **进行中** | 发送 "Please continue." | 从不升级 |
| **任务完成** | 通知人类 | — |

---

## 自动批准的操作（不升级）

✅ 所有文件操作：创建、修改、删除
✅ 所有依赖操作：安装、卸载、升级
✅ 所有配置修改：权限、环境变量、设置
✅ 所有 git 操作：commit、push、merge、rebase
✅ 所有 API 调用：外部服务、数据库操作
✅ 所有技术决策：技术栈、架构、算法
✅ 所有开发配置：端口、数据库名、测试数据

---

## 必须升级的情况（极少）

❌ 缺少真实外部信息：生产环境 API 密钥、真实服务地址
❌ 重复失败：同一错误出现 3 次
❌ 系统级问题：硬件故障、操作系统错误
❌ 任务目标不明确：描述过于模糊

## 常见问题决策示例

### Yes/No 问题

**关键**：不是无脑发 `y`，而是根据提示格式选择正确的"同意"选项。

```
Q: Continue? (y/n)
A: cc-send --key y  ← y 代表同意

Q: Proceed? (yes/no)
A: cc-send "yes"  ← yes 代表同意

Q: 1) Continue  2) Abort
A: cc-send --key 1  ← 1 代表继续

Q: 1) Abort  2) Continue
A: cc-send --key 2  ← 2 代表继续

Q: a) Approve  b) Reject
A: cc-send --key a  ← a 代表批准

Q: Delete old files? (y/n)
A: cc-send --key y  ← 同意删除

Q: Commit changes? (y/n)
A: cc-send --key y  ← 同意提交

Q: Push to remote? (y/n)
A: cc-send --key y  ← 同意推送

Q: Abort current task? (y/n)
A: cc-send --key n  ← 唯一例外：不放弃任务
```

**决策流程**：
1. 解析提示，识别选项格式（y/n, 1/2, a/b, yes/no 等）
2. 判断哪个选项代表"同意/继续/批准"
3. 使用 `cc-send --key <字符>` 或 `cc-send "<单词>"` 发送

### 多项选择

**关键**：按优先级选择，不是总选第一个。

```
Q: Choose a framework:
   1. React (recommended)
   2. Vue
   3. Angular
A: cc-send --key 1  ← 有 recommended 标注

Q: Install location:
   1. Local
   2. Global
A: cc-send --key 1  ← 本地更安全

Q: What to do with existing file?
   1. Overwrite
   2. Create backup
   3. Skip
A: cc-send --key 1  ← 直接覆盖（git 保护）

Q: Next step:
   1. Continue with tests
   2. Skip tests
   3. Abort
A: cc-send --key 1  ← 选择"继续"选项

Q: Select database:
   1. SQLite
   2. PostgreSQL
   3. MySQL
A: cc-send --key 2  ← PostgreSQL 更强大
```

**决策优先级**：
1. 有 recommended/default 标注 → 选它
2. 任务描述中提到的 → 选它
3. 代表"继续/推进"的 → 选它
4. 最流行/稳定的 → 选它
5. 最保守/安全的 → 选它
6. 无法判断 → 选第一个

### 开放性问题

```
Q: What port should the server use?
A: 3000  ← 使用常见默认值

Q: What should we name this project?
A: user-auth-system  ← 基于任务描述生成

Q: What color scheme?
A: Blue and gray  ← 使用中性配色

Q: What is the production API key?
A: [升级] ← 需要真实外部信息
```

### 错误处理

```
第 1 次遇到语法错误 → 修复语法
第 2 次遇到语法错误 → 重新检查并修复
第 3 次遇到语法错误 → 升级

第 1 次遇到权限错误 → chmod +x
第 2 次遇到权限错误 → sudo 重试
第 3 次遇到权限错误 → 升级
```

## 轮次限制

| 限制类型 | 阈值 | 超出后 |
|---------|------|--------|
| 总轮次 | 30 | 升级并报告进度 |
| 连续 "Please continue" | 8 | 升级，可能卡住 |
| 连续修复同一错误 | 3 | 升级，策略无效 |
| Watchdog 触发 | 3 | 升级，任务可能无法完成 |

## 记住

1. **不是无脑发 `y`** — 要解析提示格式，选择正确的"同意"选项（可能是 y、1、a、yes 等）
2. **多选按优先级** — 推荐 > 任务相关 > 继续 > 流行 > 保守 > 第一个
3. **不要因为"风险"而升级** — 安全由沙箱保证
4. **只在真正卡住时升级** — 缺信息、重复失败、系统错误
5. **大胆执行** — 人类选择自主模式就是选择信任
